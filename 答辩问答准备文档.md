# 基于 Mininet 的校园网项目答辩 Q&A 准备文档

本文用于帮助答辩同学提前准备老师可能提出的问题。问题覆盖基础架构、VLAN 与 IP 规划、ACL、防火墙、DMZ、VPN、前后端联动、运行环境与工具链等方面。

回答建议保持简洁、准确、有逻辑。答辩时不需要逐字背诵，但需要理解每个回答背后的技术原因。

## 一、项目总体架构

### Q1：你们这个项目整体做了什么？

本项目基于 Mininet 和 Open vSwitch 构建了一个校园网仿真系统。网络结构采用核心层、汇聚层、接入层的三层架构，划分了办公、教学、宿舍、人事、财务、服务器、DMZ、访客、分校区和模拟公网等多个区域。

系统不仅实现了 VLAN 隔离、跨 VLAN 路由、ACL 访问控制和主分校区互联，还扩展了 DMZ 安全区、边界防火墙、GRE VPN 接入、外部网站访问控制和前端拓扑可视化。前端操作会通过 API 调用后端真实 Mininet 网络，因此展示结果与底层网络状态保持一致。

### Q2：为什么选择“核心-汇聚-接入”三层架构？

三层架构是校园网和企业网中比较典型的设计方式。核心层负责高速转发和全网互联，汇聚层负责区域汇聚和策略控制，接入层负责终端接入和 VLAN 划分。

这种结构层次清晰，便于扩展，也便于把不同区域的流量按业务类型进行管理。例如办公区、人事财务区、宿舍区和服务器区可以分别挂在不同汇聚或接入交换机下，后续增加新区域时只需要新增对应接入交换机和 VLAN。

### Q3：项目中有哪些主要网络区域？

项目主要划分为以下区域：

- 校园内网：包括办公、教学、宿舍、人事、财务、服务器等区域，使用 `10.0.0.0/8` 下的不同子网。
- DMZ 区：使用 `10.0.110.0/24`，部署 `vpn_srv` 和 `dmz_web`。
- 模拟公网区：使用 `203.0.113.0/24`，部署外部网站 `ext_web` 和访客客户端 `gst3` 的公网入口。
- 分校区：使用 `10.1.0.0/24`，通过 `cs2`、`as4`、`sw17` 接入主校区。
- VPN 隧道网段：使用 `172.16.0.0/30`，用于 GRE 隧道内层通信。

### Q4：你们项目和普通 Mininet 拓扑相比有什么不同？

普通 Mininet 拓扑通常只展示主机和交换机之间的连通性。本项目在此基础上增加了多个工程化能力：

- 通过 VLAN 和 Trunk 实现真实二层隔离。
- 通过 Linux VLAN 子接口实现跨 VLAN 三层路由。
- 通过 OVS OpenFlow 流表实现细粒度 ACL。
- 通过 iptables 实现有状态边界防火墙。
- 引入 DMZ 和模拟公网区，支持出入站安全控制。
- 通过 GRE 隧道实现 VPN 接入。
- 通过 REST API 与前端拓扑联动，实现真实测试结果可视化。
- 通过观测面板展示 RTT、吞吐、ACL 命中数和防火墙计数。

## 二、VLAN、IP 与路由

### Q5：为什么要划分 VLAN？

VLAN 用于在二层网络中隔离不同业务区域。校园网中不同部门、宿舍、服务器和访客网络具有不同安全需求，如果全部处于同一二层广播域，会带来广播扩散、安全边界不清晰和访问控制困难等问题。

通过 VLAN 划分后，同一 VLAN 内主机可以直接二层通信，不同 VLAN 之间必须经过三层网关和访问控制策略，从而方便进行区域隔离和权限管理。

### Q6：IP 地址是按照什么原则分配的？

IP 地址按照“区域对应独立网段”的原则分配。每个 VLAN 对应一个独立 `/24` 子网，网关地址通常使用该网段的 `.1`。

例如：

- 宿舍 A：VLAN 10，网段 `10.0.10.0/24`，网关 `10.0.10.1`。
- 宿舍 B：VLAN 11，网段 `10.0.11.0/24`，网关 `10.0.11.1`。
- 办公楼：VLAN 20，网段 `10.0.20.0/24`，网关 `10.0.20.1`。
- 人事处：VLAN 50，网段 `10.0.50.0/24`，网关 `10.0.50.1`。
- 财务处：VLAN 60，网段 `10.0.60.0/24`，网关 `10.0.60.1`。
- 服务器区：VLAN 100，网段 `10.0.100.0/24`，网关 `10.0.100.1`。
- DMZ：VLAN 110，网段 `10.0.110.0/24`，网关 `10.0.110.1`。
- 模拟公网：VLAN 250，网段 `203.0.113.0/24`，网关 `203.0.113.1`。

这样分配的优点是清晰、可读、便于定位故障，也便于根据 IP 地址快速判断主机所属区域。

### Q7：为什么模拟公网使用 `203.0.113.0/24`？

`203.0.113.0/24` 是 RFC 5737 中保留的文档示例地址段，常用于教学、文档和实验场景。它不会与真实公网地址混淆，适合用来模拟 Internet 区域。

### Q8：跨 VLAN 通信是怎么实现的？

跨 VLAN 通信通过 Linux root namespace 中的虚拟路由器实现。项目创建了一对 veth 虚拟网卡：

```text
vrouter-vr0 ↔ fw1-vr0
```

其中 `fw1-vr0` 接入 OVS 防火墙节点 `fw1`，`vrouter-vr0` 留在 Linux root namespace。然后在 `vrouter-vr0` 上创建多个 VLAN 子接口，例如 `vrouter-vr0.10`、`vrouter-vr0.20`、`vrouter-vr0.110` 等，每个子接口配置对应 VLAN 的网关 IP。

当主机访问其他网段时，流量先发往自己的默认网关，再由 Linux 内核进行三层转发，最后回到目标 VLAN。

### Q9：为什么没有使用外部 SDN 控制器做路由？

本项目选择使用 Linux 内核路由加 OVS 流表的混合方式。Linux 内核路由稳定、直接，适合承担三层转发；OVS 流表适合做二层转发和 ACL 控制。这样可以避免引入复杂外部控制器，同时仍然能体现 SDN 和 OpenFlow 的访问控制能力。

这种设计更轻量，也更适合课程项目演示。

## 三、OVS、端口与网卡

### Q10：OVS 在项目中起什么作用？

OVS 是项目中的虚拟交换机。它主要承担三个作用：

- 实现二层交换。
- 配置 Access 和 Trunk 端口，实现 VLAN 隔离和 VLAN 透传。
- 通过 OpenFlow 流表实现 ACL，例如按源 IP、目的 IP 和优先级丢弃违规流量。

### Q11：Access 端口和 Trunk 端口有什么区别？

Access 端口通常连接终端主机，只属于一个 VLAN。项目中主机连接交换机的端口会配置 `tag=<vlan_id>`，表示该主机发送和接收的流量属于该 VLAN。

Trunk 端口通常连接交换机之间的链路，可以承载多个 VLAN 的带标签流量。项目中核心、汇聚、接入交换机之间的链路会配置 `trunks=<all_vlans>`，允许多个 VLAN 通过。

### Q12：项目里为什么要注意接口名长度？

Linux 网络接口名通常有长度限制，常见限制是 15 个字符以内。Mininet 会根据节点名和端口号生成虚拟网卡名称，如果节点名太长，可能导致接口创建失败。

因此项目中使用较短的节点名，例如 `cs1`、`as1`、`sw8`、`vpn_srv`、`dmz_web` 等，以保证在 WSL/Ubuntu 环境下可以稳定创建虚拟网卡。

### Q13：veth pair 是什么？为什么要用它？

veth pair 可以理解为一根虚拟网线，两端分别是两个虚拟网卡。从一端发出的数据会从另一端收到。

项目使用 veth pair 把 OVS 网络和 Linux root namespace 中的虚拟路由器连接起来。这样 Mininet 中的 VLAN 流量可以送到 Linux 内核进行三层路由和 iptables 防火墙处理。

## 四、ACL 与访问控制

### Q14：ACL 是如何工作的？

ACL 通过 OVS OpenFlow 流表实现。项目使用 `ovs-ofctl add-flow` 向交换机下发规则，规则中包含匹配条件和动作。

例如宿舍 A 不能访问财务处，可以下发类似规则：

```bash
ovs-ofctl add-flow cs1 "priority=605,ip,nw_src=10.0.10.0/24,nw_dst=10.0.60.0/24,actions=drop"
```

含义是：如果 IP 报文的源网段是 `10.0.10.0/24`，目的网段是 `10.0.60.0/24`，则丢弃。因为这条规则优先级高于默认转发规则，所以会优先匹配并生效。

### Q15：ACL 为什么需要优先级？

OVS 流表可能同时存在多条规则。优先级用于决定先匹配哪一条。项目中 drop 规则通常设置较高优先级，默认 normal 转发规则设置较低优先级。

这样可以保证违规流量优先被丢弃，而合法流量在没有命中高优先级规则时继续按 normal 方式转发。

### Q16：项目中的 ACL 主要限制了哪些访问？

主要包括：

- 宿舍区不能访问人事、财务。
- 访客 WiFi 不能访问校园内网。
- VPN 访客不能访问人事、财务。
- 人事、财务不能访问普通外部网站。
- 外部网站不能访问普通校园内网。
- DMZ 门户不能主动访问校园内网。

### Q17：ACL 和防火墙有什么区别？

在本项目中，ACL 主要由 OVS 流表实现，偏向于无状态的快速匹配，例如根据源网段和目的网段直接 drop。

防火墙主要由 iptables 实现，能够识别连接状态，例如允许内网主动访问外网后的返回包，但拒绝外网主动进入内网。

简单来说：

- OVS ACL 适合做高速、明确的访问控制。
- iptables 防火墙适合做有状态的边界策略。

## 五、边界防火墙与 DMZ

### Q18：项目中的防火墙是怎么工作的？

项目中的 `fw1` 是边界防火墙节点，连接公网区、DMZ 区和校园内网。跨区域流量都会经过 `fw1`。

具体策略由两部分组成：

- Linux root namespace 中的 iptables 负责有状态防火墙规则。
- `fw1` 上的 OVS 高优先级流表负责边界 ACL，尤其是 ICMP 入站防护。

iptables 通过 `CAMPUS_EDGE_FILTER` 链控制出入站流量，例如允许外网访问 DMZ 门户、拒绝外网进入内网、拒绝 DMZ 主动访问内网等。

### Q19：什么是三臂边界防火墙？

三臂边界防火墙指防火墙连接三个安全方向：

```text
公网 sw18 ─┐
           ├─ fw1 ─ cs1 内网
DMZ sw15 ─┘
```

三个方向分别是公网、DMZ 和内网。这样设计可以让不同安全区域之间的流量都经过统一检查点，便于实施边界访问控制。

### Q20：为什么要引入 DMZ？

DMZ 是公网和内网之间的安全缓冲区。公开服务如果直接放在内网中，外部访问会带来较大风险。放到 DMZ 后，外部可以访问公开服务，但不能直接进入内网。

项目中 `dmz_web` 作为校园公网门户部署在 DMZ 中。外部网站可以访问它，但 `dmz_web` 不能主动访问普通内网，从而体现了 DMZ 的隔离效果。

### Q21：DMZ 里有哪些节点？

DMZ 主要包括：

- `sw15`：DMZ 交换机。
- `vpn_srv`：VPN 服务器，IP 为 `10.0.110.10`。
- `dmz_web`：校园公网门户，IP 为 `10.0.110.20`。

### Q22：外部网站能访问哪些校园资源？

外部网站只能访问 DMZ 中被允许发布的服务，例如 `dmz_web` 的 HTTP 服务和 Ping 测试。外部网站不能访问普通内网主机，如办公楼、宿舍、服务器区 Web 服务等。

这体现了“公开服务放在 DMZ，普通内网默认不对外暴露”的安全原则。

### Q23：什么是 DMZ 横向移动防护？

横向移动指攻击者控制某台服务器后，继续尝试访问内网其他主机。项目中即使 `dmz_web` 被视为已被控制，它主动访问内网也会被防火墙阻断。

这说明 DMZ 不只是“放公开服务的区域”，还承担了限制风险扩散的作用。

## 六、VPN 与 GRE

### Q24：VPN 的基本原理是什么？

VPN 的作用是在不可信网络上建立一条逻辑上的安全通道。项目中使用 GRE 隧道模拟 VPN 数据面。`gst3` 是远程访客客户端，`vpn_srv` 是 VPN 服务器。

VPN 开启后，`gst3` 访问校园内网的流量会先封装进 GRE 隧道，发送到 `vpn_srv`，再由 VPN 服务器解封装后进入校园内网。

### Q25：项目中的 VPN 路径是什么？

VPN 关闭时：

```text
gst3 → sw18 → fw1 ×
```

VPN 开启后：

```text
gst3 → sw18 → fw1 → sw15 → vpn_srv → sw15 → fw1 → cs1 → 校园内网
```

这个路径体现了远程访客从公网进入 DMZ，再通过 VPN 服务器进入校园内网的过程。

### Q26：项目中的 VPN 有没有加密？

没有。项目使用的是 GRE 隧道，GRE 主要提供封装功能，不提供加密、认证和密钥协商。

因此本项目中的 VPN 更准确地说是“基于 GRE 的隧道接入模拟”，用于展示远程接入路径、路由切换和访问控制。如果要达到生产环境 VPN 安全标准，还需要结合 IPsec、SSL VPN、WireGuard 或 OpenVPN 等具备加密和认证能力的技术。

### Q27：GRE 隧道的关键配置有哪些？

关键配置包括：

- 隧道设备：`gre1`
- 客户端隧道 IP：`172.16.0.2`
- 服务端隧道 IP：`172.16.0.1`
- 隧道网段：`172.16.0.0/30`
- GRE key：`42`

VPN 开启时，客户端添加指向校园内网的路由；VPN 关闭时，删除该路由。

### Q28：VPN 开启后是否可以访问所有内网资源？

不可以。VPN 用户按宿舍类权限处理，可以访问普通校园资源和服务器区，但不能访问人事、财务等敏感区域。

这样设计体现了最小权限原则，即远程接入只获得必要权限，而不是完全放开内网。

### Q29：什么是 fail-closed？项目中如何体现？

fail-closed 指系统异常或 VPN 未建立时默认拒绝访问，而不是退回到不安全的直连路径。

项目中 VPN 关闭或 GRE 隧道不可用时，`gst3` 的物理网段不能直接访问校园内网。只有 GRE 隧道建立并配置路由后，访问普通内网才会被允许。

## 七、前后端与可视化

### Q30：前后端是否统一？

是统一的。前端 `topology_data.js` 中的节点、VLAN、链路和 ACL 规则与后端 `campus_network.py` 的配置保持对应。前端发起测试时，会通过 API 调用后端真实 Mininet 命令，而不是只做浏览器本地模拟。

此外，项目还通过 `test_topology_logic.js` 检查前端拓扑逻辑，避免出现不存在的链路或错误路径。

### Q31：前端路径高亮是怎么来的？

用户在前端选择源节点和目标节点后，前端调用 API。后端在 Mininet 中执行真实 Ping 或服务访问测试，并返回测试结果。前端根据结果和逻辑路径进行高亮：

- 成功路径显示绿色。
- 被阻断路径显示红色。
- 阻断点显示红色叉号。

例如外部网站访问内网时，阻断点通常显示在 `fw1`。

### Q32：为什么说前端不是静态拓扑图？

因为前端不仅展示拓扑，还能触发真实测试，并根据后端返回结果动态更新路径、状态、日志和观测指标。它相当于底层 Mininet 网络的可视化控制台。

### Q33：网络运行观测面板展示了什么？

观测面板展示真实运行指标，包括：

- 跨校区 Ping RTT、丢包率、抖动。
- 跨校区接口 RX/TX 速率。
- DMZ 发布与隔离探测结果。
- iptables 防火墙 DROP 计数。
- OVS ACL 命中数。
- VPN 数据面状态。

这些数据来自后端真实命令和系统计数器，不是前端写死的数据。

## 八、工具、环境与运行

### Q34：项目使用了哪些主要工具？

主要工具包括：

- Mininet：构建虚拟网络拓扑。
- Open vSwitch：实现 VLAN、Trunk、OpenFlow ACL。
- Linux veth pair：连接 OVS 与 Linux 路由命名空间。
- Linux VLAN 子接口：实现不同 VLAN 的网关。
- iptables：实现有状态防火墙策略。
- GRE tunnel：实现 VPN 隧道模拟。
- Python 标准库 HTTPServer：实现后端 API 服务。
- D3.js：绘制前端拓扑图。
- HTML/CSS/JavaScript：实现前端控制台。
- Node.js：运行前端逻辑回归测试。

### Q35：为什么在 Windows 上还要用 WSL/Ubuntu？

Mininet、OVS、Linux veth、iptables、GRE 隧道等功能依赖 Linux 网络内核能力。Windows 本身不能直接完整运行 Mininet，因此项目通过 WSL/Ubuntu 提供 Linux 网络环境，再在 Windows 浏览器中访问前端界面。

### Q36：为什么不用 Flask 或 FastAPI？

项目 API 服务主要使用 Python 标准库实现，减少额外依赖，便于在课程环境中直接运行。虽然报告中有时会说“FastAPI 风格 HTTP API”，但实际实现更接近轻量级 REST API 服务，使用标准库 HTTPServer 提供接口。

### Q37：项目如何启动？

通常可以使用项目提供的脚本：

- `run_demo.bat`：在 Windows 环境下一键启动演示流程。
- `run_mininet.sh`：启动 Mininet 后端。
- `start_api.sh`：启动 API 服务。
- `run_frontend.bat`：启动前端服务。

启动后在浏览器访问前端页面，通常是 `http://localhost:8000`。

### Q38：为什么有些拓扑修改需要重启 Mininet？

新增节点、链路、OVS 端口、VLAN 子接口、iptables 规则和 GRE 隧道等都属于运行时网络状态。Mininet 启动后拓扑结构已经创建，如果修改了底层拓扑代码，需要重启 Mininet 才能重新创建这些虚拟网络对象。

## 九、测试与验证

### Q39：如何验证同 VLAN 通信？

可以执行：

```bash
dA1 ping -c 4 dA2
```

两台主机都属于宿舍 A 的 VLAN 10，理论上应二层直接通信，结果应为 0% packet loss。

### Q40：如何验证跨 VLAN 路由？

可以执行：

```bash
dA1 ping -c 4 ofc1
```

`dA1` 属于宿舍 A，`ofc1` 属于办公楼。它们处于不同 VLAN，需要经过 Linux 虚拟网关完成三层转发。若配置正确，应能 Ping 通。

### Q41：如何验证 ACL 生效？

可以执行：

```bash
dA1 ping -c 4 hr1
```

宿舍访问人事处应被阻断，结果应为 100% packet loss。然后可以查看 OVS 流表命中数：

```bash
ovs-ofctl dump-flows cs1
```

对应 drop 规则的 `n_packets` 应增加。

### Q42：如何验证外网不能访问内网？

可以执行：

```bash
ext_web ping -c 4 ofc1
```

外部网站访问普通内网应被 `fw1` 阻断，结果应为 100% packet loss。可以查看 `fw1` 的 OVS 流表或 iptables 计数器。

### Q43：如何验证外网可以访问 DMZ 门户？

可以执行：

```bash
ext_web ping -c 4 dmz_web
ext_web curl http://10.0.110.20
```

这两个测试应成功，因为 DMZ 门户是对外发布的校园公开服务。

### Q44：如何验证 VPN 功能？

可以先关闭 VPN，测试：

```bash
gst3 ping -c 4 dB1
```

应失败。然后开启 VPN，再测试同样命令，应成功。最后测试：

```bash
gst3 ping -c 4 hr1
```

即使 VPN 开启，也应失败，因为 VPN 用户不能访问人事处。

## 十、创新点与答辩亮点

### Q45：如果老师问项目创新点，应该怎么回答？

可以从四个方面回答：

第一，项目不仅实现普通 VLAN 和 ACL，还引入了 DMZ、边界防火墙和模拟公网区，形成“公网-DMZ-内网”的安全分区架构。

第二，项目使用三臂边界防火墙 `fw1` 统一管控公网、DMZ 和内网之间的流量，使拓扑展示、真实转发路径和安全策略保持一致。

第三，项目结合了 OVS 流表和 iptables 有状态防火墙，既能做高性能 ACL，又能根据连接状态区分“内网主动访问外网的返回包”和“外网主动进入内网的新连接”。

第四，项目实现了前后端真实联动和运行观测面板，前端展示的数据来自 Mininet 后端真实测试，而不是静态模拟。

### Q46：项目最大的闪光点是什么？

项目最大的闪光点是把多个网络知识点整合成了一个完整闭环：Mininet 负责真实网络，OVS 负责 VLAN 和 ACL，Linux 负责路由和防火墙，GRE 负责 VPN，前端负责可视化，API 负责把操作和真实网络连接起来。

因此它不仅能说明“网络如何连通”，还能展示“网络如何被安全地控制和观测”。

### Q47：相比只画拓扑图，你们项目的优势是什么？

本项目的拓扑图与底层网络状态联动。前端发起测试后，后端在 Mininet 中执行真实命令，返回真实结果，再由前端进行路径高亮和阻断显示。

因此拓扑图不是静态示意图，而是网络运行状态的可视化表达。

### Q48：相比只写 Mininet 后端，你们项目的优势是什么？

只写 Mininet 后端虽然能验证网络连通性，但演示不够直观。本项目增加了 Web 前端和观测面板，可以让老师和同学直观看到拓扑结构、访问路径、阻断点、VPN 状态和运行指标，更适合课程答辩和教学展示。

## 十一、可能的追问与简短回答

### Q49：为什么人事财务要限制访问外网？

人事和财务属于敏感部门，包含隐私或财务数据。限制其访问普通外部网站可以降低数据泄露和外部攻击风险。同时项目通过白名单允许其访问 DMZ 校园门户，兼顾安全和必要业务访问。

### Q50：为什么访客 WiFi 要隔离？

访客网络的用户身份和设备安全状态不可控，因此应与校园内网隔离。项目中访客 WiFi 只允许有限范围内互访，不允许访问内网资源。

### Q51：为什么需要 DMZ，而不是直接把门户放内网？

如果公开门户直接放在内网，外部访问会扩大内网暴露面。放在 DMZ 后，即使门户被攻击，也可以通过防火墙阻断其继续访问内网，从而降低风险。

### Q52：为什么要同时使用 iptables 和 OVS 流表？

iptables 擅长有状态策略，例如识别返回流量；OVS 流表擅长高速匹配和转发控制。两者结合可以同时满足边界状态控制和高性能 ACL 的需求。

### Q53：如果 GRE 没有加密，为什么还用它？

因为本项目重点是展示 VPN 的隧道路径、路由切换和访问控制机制。GRE 实现简单、依赖少，适合课程演示。如果面向真实生产环境，应在 GRE 外叠加 IPsec，或使用 WireGuard、OpenVPN 等具备加密认证能力的方案。

### Q54：你们的项目是否是真实网络？

它是运行在单机上的网络仿真系统，但使用的技术是真实 Linux 网络技术，包括 OVS、veth、VLAN 子接口、iptables、GRE 和真实 Ping/curl 命令。因此它不是物理网络，但具备真实网络协议栈和真实转发行为。

### Q55：项目还有哪些可以改进的地方？

可以继续扩展动态路由协议、DHCP、NAT、真实外网接入、防火墙高可用、日志审计、应用层访问控制，以及更完整的 VPN 加密认证机制。

## 十二、答辩表达建议

### 1. 回答架构问题时

建议按“分层结构 + 安全区域 + 数据路径”三步回答。

例如：先说明核心-汇聚-接入，再说明公网、DMZ、内网三区，最后说明跨区流量经过 `fw1`。

### 2. 回答技术问题时

建议说明“用了什么技术、解决什么问题、项目中如何体现”。

例如回答 ACL：用了 OVS OpenFlow 流表，解决区域访问控制问题，项目中用高优先级 drop 规则阻断宿舍访问人事财务。

### 3. 回答创新点时

不要只说“功能多”，应强调体系化设计：

- 安全区域划分。
- 三臂边界防火墙。
- DMZ 隔离。
- 有状态防火墙。
- VPN 最小权限。
- 前后端真实联动。
- 运行观测能力。

可以用正面表达：

“目前项目重点放在网络架构、访问控制和可视化演示上。后续可以进一步扩展生产级 VPN 加密、动态路由、防火墙高可用和更细粒度的应用层安全策略。”
