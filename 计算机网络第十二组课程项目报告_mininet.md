报告内容

一、项目要求 
本计算机网络课程项目“基于 Mininet 的校园网构建”旨在利用 Mininet 网络仿真平台与 Open vSwitch（OVS）虚拟交换机，构建一个覆盖学生宿舍、办公楼、教学楼、人事处、财务处及服务器区等多个区域的校园网络模型。本项目综合考虑了现代校园网络的实际需求，在仿真环境中实现多区域网络的划分、互通与隔离，并配套开发了前端拓扑可视化系统。此外，为模拟多校区及远程安全访问的真实场景，项目还要求实现主分校区专线互联，以及基于 GRE 协议的 VPN 隧道安全接入功能。

二、实现原理及方案
1. 项目调研方案 
为了设计出符合现代高校运作规范的校园网络，项目组在前期进行了充分的技术调研和架构对比分析。
在网络拓扑结构的选择上，本项目组调研了扁平化网络与三层分层网络的优缺点。最终决定采用Cisco经典的“核心-汇聚-接入”三层模型。核心层负责高速数据交换和三层路由；汇聚层负责策略控制、VLAN终结和区域流量汇聚；接入层负责最终用户设备的网络接入和VLAN标签的打入。这种架构不仅结构清晰，而且非常利于未来的规模扩展。
通过调研，发现Mininet 中实现 VLAN 间路由常见方案包括引入外部 SDN 控制器统管流表，或在宿主机侧配置 Linux 路由。本项目主程序摒弃了依赖外部 SDN 控制器的常规做法，创新性地采用了一种贴近现代工业界白盒交换机的轻量化混合数据面架构（Hybrid Architecture）：在 Linux root 命名空间创建 Veth Pair（vrouter-vr0 ↔ cs1-vr0），通过 802.1Q VLAN 子接口配置各网段网关，结合内核 IP 转发实现跨 VLAN 通信；而安全控制（ACL 防火墙）则通过 `ovs-ofctl` 向核心交换机精准下发静态 OpenFlow 流表。这种“控制面与数据面解耦、路由与安全分离”的设计，充分发挥了 Linux 原生协议栈的稳定性与 OVS 硬件级转发的高效性。

2. 实现项目的关键技术 
(1) Mininet 网络虚拟化
利用 Mininet Python API（`Topo`、`Mininet`、`OVSKernelSwitch`）以代码定义网络节点与链路。通过继承 `CampusTopo(Topo)` 类，编排 2 台核心、4 台汇聚、17 台接入交换机及 21 台主机、4 台服务器的三层校园网拓扑。
(2) Open vSwitch (OVS) 虚拟交换
项目采用OVS替代Mininet默认的Linux Bridge。OVS支持工业级的VLAN划分和OpenFlow协议。项目中通过ovs-vsctl命令行工具动态配置接入交换机端口的VLAN Access模式（打Tag）以及交换机间级联端口的Trunk模式（允许指定VLAN通过）。
(3) Linux Veth Pair 与 网络命名空间路由（核心三层路由）
这是本项目实现跨网段通信的核心机制。创建 Veth Pair 虚拟链路（vrouter-vr0 ↔ cs1-vr0）：cs1-vr0 接入核心交换机 Trunk 口，vrouter-vr0 驻留于 Linux root 命名空间（非 Mininet Host 节点）。在 vrouter-vr0 上为 19 个 VLAN 各建 802.1Q 子接口并配置网关 IP（如 10.0.10.1），开启 `net.ipv4.ip_forward=1`，由 Linux 内核承担三层转发，使 cs1 成为全网路由枢纽。
(4) OpenFlow 流表与 ACL 访问控制
通过ovs-ofctl工具直接向核心交换机cs1下发OpenFlow流表规则来实现ACL防火墙功能。利用流表的优先级（Priority）机制，将丢弃规则（Action=drop）的优先级设置得比普通转发规则高。匹配字段设定为网络层的源IP段（nw_src）和目的IP段（nw_dst），从而精确地阻断了诸如“学生宿舍(10.0.10.0/24) 访问 人事处(10.0.50.0/24)”的非法流量。
除 drop 规则外，项目还配置了 priority=608 等显式放行规则；priority=1 的默认 normal 规则保障其余合法流量正常转发。drop 规则优先级（如 603—619）高于默认规则，确保 ACL 优先匹配。
(5) 基于 GRE 协议的 VPN 安全接入隧道
为了实现校园访客（如教职工远程办公）安全访问校内资源，本项目依托 Linux 原生网络功能构建了 GRE (Generic Routing Encapsulation) 隧道。通过在独立网络命名空间之间配置隧道接口（gre0），将远程终端（gst3）的流量封装后直达校内 VPN 网关（vpn_srv）。结合 OVS 下发的 ACL 流表，确保未连 VPN 时严格阻断，开启 VPN 后实现细粒度的授权访问。
(6) 基于 Web 的交互式仿真控制台与拓扑可视化
本项目的前端通过构建一个交互式仿真控制台与轻量级数字孪生平台来达成最好的演示效果。系统采用 Mininet 后端 + Web 前端双轨架构：后端在 WSL/Ubuntu 中运行提供真实网络仿真与 REST API；前端在 Windows 浏览器中利用 D3.js 构建动态拓扑。用户在前端界面触发的每一次 Ping 测试或 VPN 开关切换，均会通过 API 转化为底层真实的 ICMP 探针或隧道配置指令，并将底层真实的延迟、丢包及 OVS 流表匹配状态（如 ACL 阻断的红色 ✕ 标记及连通的绿色高亮）实时反向映射到可视化图表上。这种从指令下发到真实流量反馈的完整闭环，极大提升了网络策略的编排与测试体验。

三、系统设计过程及可行性分析 
1. 系统的总体结构 
系统的总体网络拓扑结构是一个严格的树状多层架构。全网配置了2个核心交换机，4个汇聚交换机，17个接入交换机，以及21台主机和4台服务器。
(1) 核心层：部署主校区核心交换机 cs1 和分校区核心交换机 cs2。cs1 连接了所有的主校区汇聚交换机及分校区核心，并连接了虚拟路由网关和部分直连节点（如 vpn_srv、gst3 所接接入），是整个校园网络的数据交换和路由转发枢纽。
(2) 汇聚层：部署四台汇聚交换机，按地理与业务逻辑划分区域：
as1：负责办公与教学区域（办公楼、实验室、图书馆、教学楼、人事处、财务处、后勤）。
as2：负责生活与无线区域（宿舍A/B/C区、食堂、办公WiFi、访客WiFi）。
as3：负责数据中心区域（服务器区 Server、隔离区 DMZ、管理网段）。
as4：负责分校区网络汇聚。
(3) 接入层：部署了17台接入交换机（sw1 至 sw17，包含专用于校园访客的接入 sw16），下沉到具体的楼宇和部门。每一台接入交换机被配置为一个特定的VLAN域。
(4) 终端与服务器：已在办公（ofc1—3）、教学（acd1—3）、宿舍 A/B（dA1—3、dB1—3）、人事（hr1—2）、财务（fin1—2）、访客（gst1—3）、分校区（c2h1—2）及服务器区（web、ftp、dns、vpn_srv）部署终端；各终端默认网关指向对应 VLAN 的虚拟网关 IP。

(3) 第三步：启动Mininet并配置OVS工作模式
调用self.net.start()启动网络后，以 controller=None 启动，无外部 SDN 控制器。遍历所有虚拟交换机，通过执行系统命令 ovs-vsctl set-fail-mode <sw> standalone 将它们设置为独立工作模式。这意味着在没有外部SDN控制器接入的情况下，交换机能够依靠传统的MAC地址学习机制进行二层转发。
(4) 第四步：配置接入端口与VLAN隔离 (_setup_access_ports 方法)
为了实现真正的二层隔离，代码遍历了网络中的所有链路。
对于接入端口（一端是交换机，一端是主机的端口）：通过查找主机所属的VLAN ID，使用命令 ovs-vsctl set port <port_name> tag=<vlan_id> 将该端口配置为Access模式。
对于级联端口（交换机连接交换机的端口）：使用命令 ovs-vsctl set port <port_name> trunks=<all_vlans> 将其配置为Trunk模式，允许所有业务VLAN的带标签数据帧（Tagged Frame）通过。
     
                图4  OVS 接入端口与Trunk端口配置代码

(5) 第五步：配置三层单臂路由(_setup_router 方法)
首先，在Linux系统内部执行 ip link add vrouter-vr0 type veth peer name cs1-vr0 创建一条“虚拟网线”。
循环遍历我们规划的 VLAN_CONFIG 字典。对于每一个VLAN，使用 ip link add link vrouter-vr0 name vrouter-vr0.<vid> type vlan id <vid> 创建基于802.1Q协议的VLAN子接口。
为每个子接口配置网关IP地址，例如 ip addr add 10.0.10.1/24 dev vrouter-vr0.10，并将其状态置为UP。
通过 sysctl -w net.ipv4.ip_forward=1 打开Linux内核的IPv4报文转发开关。
将Veth对的另一端 cs1-vr0 添加到核心交换机 cs1 中，并设置为允许所有VLAN通过的Trunk口。此时，各个VLAN的跨网段流量抵达核心交换机后，都会被送往Linux内核进行三层路由处理，再被送回对应的目的VLAN中。

(6) 第六步：基于GRE协议的VPN隧道配置实现
    为满足多校区跨网安全通信需求，在原有单臂路由基础上，增加GRE隧道模式的VPN功能。VPN作为跨校区安全传输通道，可对两校区之间的通信数据进行封装隔离，提升网络安全性。
本功能依托Linux系统原生隧道工具实现，核心流程为：定义VPN服务端与客户端IP地址，通过命令创建GRE隧道接口并指定隧道两端端点，配置隧道接口IP并开启系统IPv4转发，最后添加静态路由，使跨校区流量优先经由VPN隧道转发。VPN隧道完整配置代码如图7所示。

(7) 第七步：下发ACL安全访问控制流表 (_setup_acl 方法)
通过 ovs-ofctl 工具向核心交换机 cs1 写入OpenFlow静态流表。例如，为了阻止宿舍A区访问财务处，代码生成并执行了如下命令：
ovs-ofctl add-flow cs1 "priority=605,ip,nw_src=10.0.10.0/24,nw_dst=10.0.60.0/24,actions=drop"
这里指定了网络协议为ip，源网段为宿舍A，目的网段为财务处，动作为丢弃（drop）。由于默认路由转发的优先级较低（通常为默认优先级或者由系统内核处理），优先级设置为600+的丢弃规则会被优先匹配，从而成功实现了防火墙功能。

五、项目总结 
1. 设计中遇到的主要困难和解决方案。
(1) 独立模式下OVS无法实现跨网段三层路由。
我们最初尝试将交换机全部设置为 standalone 模式期望自动互通，发现不同VLAN主机即使网关正确也无法Ping通。standalone 模式下 OVS 负责二层交换，不具备跨 VLAN 三层路由能力。
解决方案：在 Linux root 命名空间动态创建 Veth Pair 与 VLAN 子接口，配置网关并开启 IP 转发，由内核承担三层路由，OVS 专注二层转发与 ACL，代码轻量且稳定。
(2) ACL规则优先级冲突导致全网断网。
在核心交换机下发 ACL 的 drop 流表时，因优先级设置有误，导致合法跨 VLAN 流量因没有明确命中规则，也被 OVS 默认拦截。
解决方案：严格规划流表优先级，将 drop 规则设为 603—619 高优先级，放行规则设为 608 等中优先级，默认 normal 规则 priority=1 兜底，确保违规流量被精确阻断。
(3) GRE隧道频繁重连导致前端控制台高延迟。
在开发前端 VPN 交互时，最初每次状态同步都会触发后台销毁并重建 GRE 隧道，这在 WSL 虚拟化环境下导致了极高的系统开销与 ICMP 延迟（往往超过 3000ms）。
解决方案：在 Python 后端 API 引入了隧道健康检测机制（Ping Probe）与状态缓存，仅在隧道真正断开或需要时才触发重建，使 VPN 开启状态下的 Ping 延迟大幅降低，保障了前端演示的流畅性。

3．项目小结与未来展望
本项目基于 Mininet 与 OVS 完成了校园网仿真：19 个 VLAN 二层隔离、Veth 单臂路由实现跨网段三层互通、OpenFlow ACL 实现宿舍区/访客区访问控制，并配套 Web 前端拓扑可视化。项目覆盖课程要求的二层/三层互通、服务器区访问与安全策略等核心功能，具备较好的演示与扩展基础。

4. 未来展望：
(1) 动态路由协议的支持
当前采用静态默认路由 + 内核转发；网络规模扩大时可引入 OSPF 等动态路由协议，支持多核心路由器场景。
(2) 外部网络接入
当前为独立仿真网段，主机使用静态IP。下一步可以加入DHCP服务器实现IP动态分配，并在虚拟路由器边缘配置NAT，使得Mininet内的虚拟主机能够真实访问外部Internet网络。
(3) SDN 控制器集成
项目仓库已含 `simple_controller.py`（Ryu L2 控制器）及 `run_mininet.sh` 备选启动脚本；后续可探索控制器统管路由与安全流表，实现更灵活的策略下发。
5. 对课程项目的意见和建议
马老师的《计算机网络》课程通过布置mininet这样兼具理论深度和工程实践的项目，在计算机网络教材之外加深了我们对网络协议栈和与 OVS 流表原理的理解。从抽象的课本概念（VLAN、路由表、ACL）到写代码实际在虚拟交换机中敲击命令配置，这样的实践给我们带来的实践能力与工程素养提升是巨大的。
