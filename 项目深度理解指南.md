# 基于 Mininet 的校园网项目深度理解指南

本项目可以理解为一个小型校园网的“可运行数字孪生系统”：后端使用 Mininet 和 Open vSwitch 构建真实可转发的虚拟网络，前端使用 Web 拓扑图展示网络结构、路径高亮和策略效果，API 层把前端操作转化为真实 Ping、HTTP、FTP、VPN 和观测指令。它不是单纯画一张网络图，也不是只写一个 Mininet 拓扑，而是把拓扑、转发、访问控制、VPN、安全边界和可视化观测串成了闭环。

## 一、项目整体定位

本项目的主题是“基于 Mininet 的校园网构建”。从课程要求看，它需要完成多 VLAN、多区域、主分校区互联、服务器访问、ACL 安全控制、VPN 远程接入等功能。从最终实现看，项目已经进一步扩展为一个带有 DMZ、边界防火墙、模拟公网和运行观测面板的校园网络安全仿真平台。

项目覆盖的典型校园网络场景包括：

- 宿舍、办公楼、教学楼、图书馆、人事处、财务处、服务器区等多区域网络划分。
- 同 VLAN 内二层互通，不同 VLAN 之间通过三层网关互通。
- 宿舍等普通区域不能访问人事、财务等敏感区域。
- 人事、财务等保密部门有更严格的外联限制。
- 主校区与分校区通过核心链路互联。
- 访客 WiFi 与校园内网隔离。
- 远程访客通过 VPN 才能访问校园普通内网资源。
- 外部网站可以访问 DMZ 中的校园门户，但不能访问普通内网。
- DMZ 门户即使被入侵，也不能主动横向访问内网。
- 前端可以实时展示测试路径、阻断位置、VPN 状态与运行指标。

如果用一句话总结：

> 这是一个把“校园网分层架构、VLAN 隔离、三层路由、ACL、防火墙、DMZ、VPN、前后端联动、运行观测”融合在一起的网络综合实践项目。

## 二、项目文件结构

项目目录中的主要文件可以按职责分为五类。

### 1. Mininet 后端拓扑与网络配置

核心文件：`campus_network.py`

这是整个项目的网络底座，负责：

- 定义 VLAN、交换机、主机、服务器、VPN 等配置。
- 创建 Mininet 拓扑。
- 配置 OVS 交换机端口的 Access/Trunk 模式。
- 创建 Linux veth pair 和 VLAN 子接口，实现三层网关。
- 配置 iptables 边界防火墙规则。
- 下发 OVS ACL 流表。
- 配置 GRE VPN 隧道。
- 启动 Web、FTP、DMZ 门户等演示服务。

理解项目时，应该优先阅读这个文件中的几个配置区：

- `VLAN_CONFIG`：所有 VLAN、网段、网关的来源。
- `SWITCHES`：核心、汇聚、防火墙交换机定义。
- `ACCESS_SW`：接入交换机与 VLAN、汇聚层的对应关系。
- `HOSTS`：普通终端主机定义。
- `SRV_HOSTS`：服务器、VPN、DMZ、公网网站定义。
- `CampusTopo`：Mininet 拓扑创建逻辑。
- 路由、ACL、VPN、服务启动相关方法：项目真实运行能力的关键。

### 2. 后端 API 服务

核心文件：`api_server.py`

这个文件负责把浏览器操作连接到 Mininet 后端。前端点击“Ping 测试”“开启 VPN”“刷新观测面板”时，并不是只在浏览器里模拟，而是通过 API 调用这里的逻辑，再由 Mininet 中的真实主机执行命令。

它主要负责：

- 执行真实 Ping 测试。
- 执行 HTTP、FTP 服务访问测试。
- 控制 VPN 开关与 GRE 隧道状态同步。
- 查询 OVS 流表、iptables 计数器、接口流量等运行指标。
- 返回前端需要展示的路径、阻断状态、日志和观测数据。

可以重点关注这些能力：

- VPN 控制：`/api/vpn-sync`
- 运行观测：`/api/observability`
- 安全指标：`/api/security-metrics`
- Ping/服务访问测试相关 API

### 3. 前端拓扑与交互界面

主要文件：

- 反馈结果：把真实结果变成绿色路径、红色阻断、阻断叉、日志和指标面板。

## 三、整体架构：从校园网到安全仿真平台

项目采用“核心-汇聚-接入”的经典三层校园网结构，并在边界增加了 DMZ 和模拟公网区域。

整体可以分为四个区域：

| 区域 | 网段 | 关键节点 | 作用 |
| --- | --- | --- | --- |
| 校园内网 | `10.0.0.0/8` | `cs1`、`as1/as2/as3`、`sw1-sw14` | 办公、宿舍、教学、服务器、敏感部门 |
| DMZ 区 | `10.0.110.0/24` | `sw15`、`vpn_srv`、`dmz_web` | VPN 接入、校园公网门户 |
| 模拟公网区 | `203.0.113.0/24` | `sw18`、`ext_web`、`gst3` | 外部网站、远程访客物理入口 |
| 分校区 | `10.1.0.0/24` | `cs2`、`as4`、`sw17`、`c2h1/c2h2` | 第二校区网络 |

项目最终形成的边界安全结构可以简化为：

```text
外部网站 ext_web ─┐
访客客户端 gst3 ──┴─ 公网边界 sw18 ─ fw1 ─ cs1 ─ 校园内网
                                      │
                                      └─ DMZ sw15
                                          ├─ VPN 服务器 vpn_srv
                                          └─ 校园公网门户 dmz_web
```

其中 `fw1` 是理解 v3.4.0 架构的关键。它不是普通交换机，而是边界安全枢纽：

- 一边连接公网区 `sw18`。
- 一边连接 DMZ 区 `sw15`。
- 一边连接校园内网核心 `cs1`。
- 所有跨安全区流量都要经过它。

这就是项目中的“三臂边界防火墙”思想。

## 四、核心数据流：一次 Ping 是怎么发生的

想真正理解项目，必须理解一次访问请求从前端到网络底层是怎么流动的。

以“前端测试 dA1 访问 hr1”为例：

1. 用户在浏览器选择源主机 `dA1`、目标主机 `hr1`，点击测试。
2. 前端 `api.js` 发送 API 请求给后端。
3. 后端 `api_server.py` 找到 Mininet 中的 `dA1` 主机。
4. 后端执行类似 `dA1 ping -c 4 hr1` 的真实命令。
5. 报文从 `dA1` 发出，进入 `sw8`，再通过 `as2`、`cs1` 进入三层转发路径。
6. 如果命中 ACL 规则，例如宿舍访问人事被禁止，`cs1` 上的 OVS drop 流表会丢弃该报文。
7. 后端解析 Ping 输出，得到丢包率、延迟、是否成功。
8. 后端把结果、路径、阻断点返回前端。
9. 前端 `topology.js` 根据结果把路径画成红色，并在阻断点显示叉号。

这说明项目中的前端高亮并不是凭空模拟，而是基于真实命令执行结果进行展示。

## 五、技术点详解

### 1. Mininet：用代码创建可运行网络

Mininet 是本项目的虚拟网络运行环境。它可以在一台机器上创建多个虚拟主机、交换机和链路，让它们像真实网络一样通信。

项目中通过 `CampusTopo(Topo)` 创建网络。核心思路是：

- `addSwitch()` 创建 OVS 交换机。
- `addHost()` 创建虚拟主机。
- `addLink()` 创建虚拟链路。
- 每个主机配置 IP 和默认网关。

Mininet 的优势是所有节点都可以执行真实 Linux 网络命令，例如：

```bash
dA1 ping dA2
ofc1 curl http://10.0.100.2
gst3 ip route show
```

所以项目能够做到“前端操作，后端真实执行”。

### 2. OVS：实现 VLAN、Trunk 与 OpenFlow ACL

Open vSwitch 是项目中的虚拟交换机。它比普通 Linux Bridge 更适合网络实验，因为它支持：

- VLAN Access 端口。
- VLAN Trunk 端口。
- OpenFlow 流表。
- 按 IP、网段、协议、优先级匹配并执行 drop/normal 等动作。

项目中的接入端口配置类似：

```bash
ovs-vsctl set port sw8-ethX tag=10
```

表示这个端口属于 VLAN 10。

交换机之间的级联端口配置为 Trunk：

```bash
ovs-vsctl set port as2-ethX trunks=10,11,12,...
```

表示多个 VLAN 可以带标签通过。

ACL 使用 OpenFlow 流表，例如：

```bash
ovs-ofctl add-flow cs1 "priority=605,ip,nw_src=10.0.10.0/24,nw_dst=10.0.60.0/24,actions=drop"
```

这条规则表示：宿舍 A 网段访问财务网段时，直接丢弃。

### 3. VLAN：校园区域隔离的基础

项目把不同校园区域划分到不同 VLAN 中，例如：

| VLAN | 区域 | 网段 |
| --- | --- | --- |
| 10 | 宿舍 A | `10.0.10.0/24` |
| 11 | 宿舍 B | `10.0.11.0/24` |
| 20 | 办公楼 | `10.0.20.0/24` |
| 50 | 人事处 | `10.0.50.0/24` |
| 60 | 财务处 | `10.0.60.0/24` |
| 100 | 服务器区 | `10.0.100.0/24` |
| 110 | DMZ | `10.0.110.0/24` |
| 250 | Internet 模拟区 | `203.0.113.0/24` |

VLAN 的意义是：同一 VLAN 内可以二层互通，不同 VLAN 默认二层隔离，需要经过三层网关才能互通。

### 4. Linux VLAN 子接口：实现三层网关

OVS 负责二层交换，但跨 VLAN 通信需要三层路由。项目没有依赖外部路由器，而是在 Linux root namespace 中创建虚拟路由器。

关键链路是：

```text
vrouter-vr0 ↔ fw1-vr0 ↔ fw1
```

`vrouter-vr0` 上会创建多个 VLAN 子接口：

```text
vrouter-vr0.10   10.0.10.1/24
vrouter-vr0.20   10.0.20.1/24
vrouter-vr0.50   10.0.50.1/24
vrouter-vr0.100  10.0.100.1/24
vrouter-vr0.110  10.0.110.1/24
vrouter-vr0.250  203.0.113.1/24
```

每个子接口就是对应 VLAN 的默认网关。主机访问其他网段时，会把报文交给自己的网关，再由 Linux 内核转发。

这种设计的亮点是：

- 不需要复杂 SDN 控制器。
- 路由逻辑稳定，依赖 Linux 内核协议栈。
- OVS 专注二层转发和 ACL。
- 防火墙策略可以集中在 root namespace 中用 iptables 执行。

### 5. 三臂边界防火墙：项目安全架构的核心

`fw1` 是项目 v3.4.0 的关键节点。它将网络分成三个安全方向：

```text
公网 sw18 ─┐
           ├─ fw1 ─ cs1 内网
DMZ sw15 ─┘
```

为什么这个设计重要？

因为真实网络中，内网、DMZ、外网不应该随意互通。公开服务应该放在 DMZ 中，让外网可以访问它，但不能因此进入内网。内网可以主动访问外部，但外部不能主动进入内网。

`fw1` 的作用就是让所有跨区域流量都经过同一个安全检查点：

- 外网到 DMZ：只允许访问公开服务。
- 外网到内网：默认拒绝。
- DMZ 到内网：默认拒绝，防止横向移动。
- 内网到外网：按部门策略决定是否允许。
- VPN 到内网：必须通过 GRE 隧道和访问控制。

这比简单的“所有交换机都接到核心”更接近真实校园网或企业网架构。

### 6. iptables 有状态防火墙

OVS 流表适合快速匹配和丢弃流量，但出入站控制还需要“状态”的概念。例如：

- 内网主动访问外网，外网返回包应该允许。
- 外网主动访问内网，应该拒绝。

这两种流量可能源/目的地址看起来相似，但含义完全不同。iptables 的 conntrack 可以识别连接状态：

- `NEW`：新连接。
- `ESTABLISHED`：已建立连接的后续流量。
- `RELATED`：与已有连接相关的流量。

项目用 `CAMPUS_EDGE_FILTER` 自定义链表达边界策略：

- 允许合法返回流量。
- 允许外网访问 DMZ 门户。
- 拒绝外网主动进入内网。
- 拒绝外网进入 VPN 隧道网段。
- 拒绝 DMZ 门户主动访问内网。

这是本项目从“简单 ACL”升级到“边界安全策略”的关键一步。

### 7. DMZ：让公网服务可访问但不暴露内网

DMZ 是本项目的一个重要创新点。它包含：

- `sw15`：DMZ 交换机。
- `vpn_srv`：VPN 服务器。
- `dmz_web`：校园公网门户。

DMZ 的策略可以概括为：

```text
外网 → DMZ 门户：允许
外网 → 内网：拒绝
DMZ 门户 → 内网：拒绝
内网 → DMZ 门户：按策略允许
```

这样就实现了一个真实安全理念：

> 公开服务可以被访问，但公开服务本身不能成为进入内网的跳板。

前端上，DMZ 被画成独立橙色区域，并通过 `fw1 ↔ sw15` 的双向箭头展示流量方向。

### 8. 模拟公网区：让出入站控制有真实对象

如果没有外网节点，项目只能测试校园内部访问控制。v3.4.0 增加了模拟公网区：

- `sw18`：公网边界交换机。
- `ext_web`：外部网站。
- `gst3`：远程访客物理入口。

这样项目可以演示更真实的场景：

- 宿舍访问外部网站成功。
- 办公访问外部网站成功。
- 人事/财务访问外部网站失败。
- 外部网站访问校园 DMZ 门户成功。
- 外部网站访问普通内网失败。
- 远程访客不开 VPN 访问内网失败。
- 远程访客开 VPN 后访问普通内网成功。

前端上，Internet 区被画成蓝色区域，包含 `gst3`、`sw18`、`ext_web`。

### 9. GRE VPN：远程访客安全入网

VPN 功能模拟了远程用户通过公网接入校园网的过程。

`gst3` 的真实物理路径是：

```text
gst3 → sw18 → fw1
```

VPN 关闭时，它只能接触边界和 DMZ，不允许直接进入内网。

VPN 开启后，逻辑路径变成：

```text
gst3
→ sw18
→ fw1
→ sw15
→ vpn_srv
→ sw15
→ fw1
→ cs1
→ 校园内网
```

底层使用 GRE 隧道：

| 项 | 值 |
| --- | --- |
| 隧道设备 | `gre1` |
| 客户端隧道地址 | `172.16.0.2` |
| 服务端隧道地址 | `172.16.0.1` |
| GRE key | `42` |
| 隧道网段 | `172.16.0.0/30` |

VPN 设计的重点不是“只要连上就全放开”，而是：

- VPN 用户可访问普通校园资源。
- VPN 用户不能访问人事、财务。
- 物理访客网段始终不能绕过 VPN 直接进入内网。
- GRE 异常时不回退为普通直连，即 fail-closed。

这个设计体现了“远程接入也要最小权限”的安全思想。

### 10. 前端拓扑：不是图，而是网络状态投影

前端拓扑由 D3.js 绘制。它最有价值的地方不是“画得好看”，而是它与后端真实网络状态保持一致。

前端主要完成：

- 节点分层布局：核心、汇聚、接入、终端。
- 安全区域绘制：DMZ 橙色区域，Internet 蓝色区域。
- 路径高亮：成功绿色，失败红色。
- 阻断点显示：在 `fw1`、`cs1`、`as1`、`sw13` 等位置显示阻断叉。
- VPN 动态路径：VPN 开关改变后路径随之变化。
- DMZ/VPN 双向箭头：展示不同方向的流量。
- 观测面板：展示真实运行指标。

理解前端时可以关注：

- `topology_data.js`：拓扑数据和 ACL 规则。
- `topology.js`：布局、路径、阻断、绘图逻辑。
- `api.js`：前后端交互。

前端中还有多个布局调试入口，用于调整展示效果。例如：

- `MAIN_AGG_POSITION_OVERRIDES`：调整 `as1/as2/as3`。
- `MAIN_ACCESS_SW_OFFSET`：整体移动 `sw1-sw14`。
- `MAIN_ACCESS_HOST_OFFSET`：整体移动 `sw1-sw14` 下挂节点。
- `MAIN_CAMPUS2_BUTTON_OFFSET`：移动主校区中的“点击分校区”按钮。
- `INTERNET_ZONE_PADDING`：调整 Internet 蓝色边框大小。
- `INTERNET_ZONE_OFFSET`：整体移动 `gst3/sw18/ext_web`。

### 11. 网络运行观测面板：从静态演示到动态观测

观测面板是项目的另一个闪光点。它把“能不能通”扩展为“网络运行状态怎么样”。

它展示的指标包括：

- 跨校区链路 RTT。
- 丢包率。
- 抖动。
- 接口 RX/TX 字节和包数。
- 接口吞吐 Mbps。
- DMZ 探测结果。
- VPN 数据面状态。
- iptables DROP 计数。
- OVS ACL 命中数。

这些指标不是前端写死的，而是后端从真实系统中采集。例如：

- Ping 结果来自真实 `ping` 命令。
- 接口吞吐来自 Linux 接口计数器差分。
- ACL 命中数来自 `ovs-ofctl dump-flows`。
- 防火墙计数来自 `iptables -L -v -n`。

这让项目具备了“可观测性”，也让演示更加可信。

## 六、重点创新点与闪光点

### 创新点 1：从普通校园网升级为安全分区校园网

普通课程项目往往只做到 VLAN、路由、ACL。本项目进一步引入了 DMZ、边界防火墙、模拟公网，形成了真实网络中常见的安全分区设计。

这使项目不只是“能通信”，还体现了“如何安全通信”。

### 创新点 2：三臂边界防火墙让拓扑与安全策略统一

`fw1` 把公网、DMZ、内网三类流量收束到同一个安全点，前端路径、后端链路和策略执行点保持一致。这是架构上的核心升级。

### 创新点 3：iptables 与 OVS 流表协同

项目没有只依赖一种技术，而是让不同技术做适合自己的事：

- Linux 内核负责三层路由。
- iptables 负责有状态边界防火墙。
- OVS 负责 VLAN、二层转发和高优先级 ACL。
- GRE 负责 VPN 隧道封装。
- 前端负责可视化和交互。

这种组合非常接近真实工程中的“多组件协同”。

### 创新点 4：DMZ 横向移动防护

很多演示只会说“外网不能进内网”。本项目进一步模拟了“DMZ 服务被攻击后，是否能继续攻击内网”的场景，并通过策略阻断 DMZ 门户主动访问内网。

这是一个很有安全深度的设计点。

### 创新点 5：VPN 不是简单开关，而是完整安全路径

VPN 路径经过公网、边界防火墙、DMZ、VPN 服务器、再回到防火墙和内网。这个路径在前端和后端都能体现。

VPN 还具有权限边界：能访问普通内网，但不能访问人事财务。这比“开 VPN 就全网放通”更合理。

### 创新点 6：前后端真实联动

项目的前端不是静态模拟器。用户点击测试后，后端会在 Mininet 主机中执行真实网络命令，前端再根据真实结果绘制路径。这种闭环是项目演示效果最强的部分之一。

### 创新点 7：运行观测面板提升可信度

观测面板让项目从“功能演示”升级为“运行状态展示”。它可以证明链路质量、吞吐、安全计数、VPN 状态确实在变化。

这使项目更像一个小型网络运维平台。

## 八、典型演示路线

如果要向老师或同学演示，推荐按下面顺序：

1. 展示整体拓扑，说明核心-汇聚-接入三层架构。
2. 展示 VLAN 划分，说明不同区域的网段与接入交换机。
3. 测试同 VLAN 与跨 VLAN 通信，证明基础网络可用。
4. 测试宿舍访问人事/财务失败，证明内网 ACL 生效。
5. 展示 Internet 区和 DMZ 区，说明 `fw1` 三臂防火墙。
6. 测试外网访问 DMZ 门户成功。
7. 测试外网访问内网失败。
8. 测试 DMZ 门户访问内网失败，说明横向移动防护。
9. 测试人事/财务访问外网失败，但访问 DMZ 门户成功，说明白名单策略。
10. 关闭 VPN 测试 `gst3` 访问内网失败。
11. 开启 VPN 测试 `gst3` 访问普通内网成功。

这条路线能完整展示项目从基础网络到安全架构再到可观测性的层层递进。

## 九、项目中容易混淆的点

### 1. fw1 是交换机还是防火墙

在 Mininet 中，`fw1` 是 OVS 交换机节点；在架构意义上，它承担边界防火墙角色。实际有状态策略由 Linux root namespace 的 iptables 执行，OVS 流表负责边界高优先级 ACL。

所以可以理解为：

> fw1 是防火墙位置和流量汇聚点，iptables 是有状态防火墙策略执行者，OVS flow 是高速边界 ACL 执行者。

### 2. DMZ 和服务器区不是一回事

服务器区 `server` VLAN 100 是校园内网服务器区，包含 `web`、`ftp`、`dns`。

DMZ VLAN 110 是边界隔离区，包含 `vpn_srv`、`dmz_web`。

公网可以访问 DMZ 门户，但不能访问内网服务器区。

### 3. gst3 不再是普通校园访客接入

`gst3` 是远程 VPN 客户端，它物理上接入公网边界 `sw18`，不是直接接入 `cs1` 或 `vpn_srv`。

它能进入内网的前提是 VPN 开启，并且流量经过 GRE 隧道。

### 4. VPN 开启不代表拥有所有权限

VPN 用户按宿舍类权限处理，可以访问普通内网资源，但不能访问人事、财务。

这是最小权限原则。

### 5. 前端路径不是随便画的

前端路径要与 `topology_data.js` 中的链路和后端 API 返回的测试结果对应。项目中专门用 `test_topology_logic.js` 防止出现虚假连线和错误路径。

## 十、总结

本项目的价值不只在于“用 Mininet 搭了一个校园网”，而在于它把多个网络课程知识点组织成了一个可运行、可测试、可展示、可观测的完整系统。

从基础层面看，它覆盖了 VLAN、Trunk、三层路由、ACL、主分校区互联等课程核心内容。

从工程层面看，它使用 Mininet、OVS、Linux veth、iptables、GRE、REST API、D3.js 等技术完成了真实联动。

从安全层面看，它引入了 DMZ、边界防火墙、出入站控制、横向移动防护、VPN 最小权限等更接近真实网络的设计。

从展示层面看，它提供了前端拓扑、路径高亮、阻断标记、VPN 状态、运行观测面板等功能，让复杂网络行为变得直观可见。

因此，这个项目适合作为计算机网络课程综合实践案例，也适合作为理解企业网络安全架构、网络仿真和前后端可视化联动的入门项目。
