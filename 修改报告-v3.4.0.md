# 校园网络模拟器修改报告

## 1. 报告概述

- 最终版本：`v3.4.0`
- 修改范围：Mininet 后端、FastAPI 风格 HTTP API、前端拓扑、访问控制、VPN、DMZ、Internet 模拟区、实时观测面板及回归测试
- 环境约束：未删除或修改既有 WSL、Ubuntu 环境配置；主要修改集中在 `mininet` 项目目录
- 核心目标：将 DMZ 从静态展示概念升级为前端路径、后端拓扑、安全策略和观测指标相互对应的主要创新点

## 2. 最终网络安全模型

最终采用简化的课程演示模型：

```text
外部网站 ext_web ─┐
访客客户端 gst3 ──┴─ 公网边界 sw18 ─ fw1 ─ cs1 ─ 校园内网
                                      │
                                      └─ DMZ sw15
                                          ├─ VPN 服务器 vpn_srv
                                          └─ 校园公网门户 dmz_web
```

安全区划分如下：

| 安全区 | 网段 | 主要节点 | 用途 |
| --- | --- | --- | --- |
| 校园内网 | `10.0.0.0/8` | `cs1`、各汇聚与接入交换机 | 办公、宿舍、服务器及保密部门 |
| DMZ | `10.0.110.0/24` | `sw15`、`vpn_srv`、`dmz_web` | VPN 接入和校园公网服务 |
| 模拟公网 | `203.0.113.0/24` | `sw18`、`ext_web`、`gst3` | 外部网站与公网 VPN 客户端 |
| VPN 隧道 | `172.16.0.0/30` | `gre1` | 访客通过 VPN 进入校园网的数据面 |

## 3. DMZ 架构升级

### 3.1 增加独立边界防火墙节点

新增 `fw1`，将原先容易被理解为“公网直接接入核心”的结构改为：

```text
sw18 ─ fw1 ─ cs1
         │
       sw15
```

最终物理连接关系：

- `sw18 → fw1`：公网安全区
- `sw15 → fw1`：DMZ 安全区
- `fw1 → cs1`：校园内网方向

DMZ 不再挂在普通服务器汇聚交换机 `as3` 下，公网边界 `sw18` 也不再直接连接 `cs1`。

### 3.2 将三层网关和策略执行点迁移到防火墙

原三层路由 trunk 使用 `cs1-vr0`。最终改为：

```text
vrouter-vr0 ↔ fw1-vr0 ↔ fw1
```

全部 VLAN 子接口仍建立在 root namespace，但 trunk 终结在 `fw1`。跨 VLAN 和跨安全区流量由 `fw1` 所在的路由与 `iptables` 策略点处理，避免出现“前端经过防火墙、后端实际绕过”的情况。

### 3.3 校园公网门户发布

在 DMZ 中增加校园公网门户：

- 节点：`dmz_web`
- 地址：`10.0.110.20`
- 服务：HTTP 80

最终策略：

- 外部网站可通过 HTTP 访问校园门户。
- 外部网站可 Ping 校园门户，作为公开服务可达性演示。
- 外部网站不得访问普通校园内网资源。
- 校园门户即使被控制，也不得主动横向访问内网或 VPN 隧道网段。

这形成了“服务可以被公网访问，但服务本身不具备进入内网的权限”的 DMZ 隔离效果。

## 4. VPN 接入改造

### 4.1 访客 3 的真实接入位置

`gst3` 从原有隐藏的校园侧接入改为真实挂载在公网边界 `sw18`。

前后端最终一致的物理入口：

```text
gst3 → sw18 → fw1
```

前端将 `gst3` 放置在公网边界正上方，不再显示不存在的 `gst3 → cs1` 或 `gst3 → VPN服务器` 直连线。

### 4.2 VPN 关闭路径

VPN 关闭时：

```text
gst3 → sw18 → fw1 ×
```

`gst3` 可以接触 DMZ 的 VPN 接入点和校园门户，但不能直接进入校园内网。

### 4.3 VPN 开启路径

VPN 开启后的完整逻辑路径：

```text
gst3
→ sw18
→ fw1
→ sw15
→ vpn_srv
→ sw15
→ fw1
→ cs1
→ 校园内网
```

后端继续使用 GRE：

- 客户端隧道地址：`172.16.0.2`
- 服务端隧道地址：`172.16.0.1`
- GRE key：`42`
- 隧道设备：`gre1`

VPN 开启后，客户端访问内网的路由指向 GRE 隧道；VPN 关闭后删除相应内网路由。GRE 不可用时保持 fail-closed，不会回退为普通直连。

### 4.4 VPN 权限

VPN 访客权限按宿舍类网络处理：

- 可访问普通校园资源和服务器区。
- 不可访问人事、财务等敏感区域。
- 物理访客网段 `10.0.91.0/24` 始终不能直接进入内网。
- 只有 GRE 内层源地址能够按 VPN 策略进入校园网。

## 5. 边界防火墙与 ACL

### 5.1 有状态边界防火墙

建立自定义链：

```text
CAMPUS_EDGE_FILTER
```

并插入 Linux `FORWARD` 链首部。

主要规则：

1. 允许符合策略的 `ESTABLISHED,RELATED` 返回流量。
2. 允许公网访问 DMZ 校园门户的 HTTP 80。
3. 允许公网 Ping DMZ 校园门户。
4. 拒绝公网主动 Ping `10.0.0.0/8` 内网。
5. 拒绝公网向内网发起新连接。
6. 拒绝公网进入 `172.16.0.0/30` VPN 隧道网段。
7. 拒绝被控制的 `dmz_web` 主动访问内网和 VPN 隧道。
8. 允许 `gst3` 接触 DMZ VPN 接入点，但拒绝其物理网段直达内网。

### 5.2 修复公网 Ping 内网仍然成功

曾出现：

```text
外部网站 → 办公楼-3：成功
```

原因是 ICMP 可能优先命中已有 conntrack 状态。最终增加两层保护：

- `iptables` 在 `ESTABLISHED,RELATED` 之前拒绝公网发起的 ICMP Echo Request。
- `fw1` 增加 OVS 高优先级规则：
  - `priority=641`：允许公网 Ping `dmz_web`
  - `priority=640`：拒绝公网 Ping 其他 `10.0.0.0/8` 地址

这样既保留“内网主动 Ping 外网”的返回包，又阻止“外网主动 Ping 内网”。

### 5.3 原有校园 ACL 保留

没有取消原有区域隔离逻辑，主要包括：

- 宿舍区不能主动访问人事、财务。
- 人事、财务可主动访问宿舍，ICMP 回包可以返回。
- VPN 访客不能访问人事、财务。
- 人事、财务禁止访问普通外部网站。
- 人事、财务可通过白名单访问 DMZ 校园门户。
- 宿舍、办公等非保密区域可以主动访问外部网站。

## 6. 外部网站与出入站模型

新增模拟外部网站：

- 节点：`ext_web`
- 地址：`203.0.113.10`
- 接入：`ext_web → sw18 → fw1`

最终方向策略：

| 场景 | 结果 |
| --- | --- |
| 宿舍/办公 → 外部网站 | 允许 |
| 人事/财务 → 外部网站 | 拒绝 |

| 外部网站 → DMZ 校园门户 HTTP | 允许 |

| DMZ 门户 → 普通内网 | 拒绝 |

## 7. 前端拓扑与路径展示

### 7.1 主拓扑整理

主要调整：

- 增加独立的 `fw1` 防火墙节点。
- 将 DMZ 整体放在拓扑右侧安全区。
- 将外部网站放入独立 Internet 区域。
- 将 `gst3` 放在公网边界 `sw18` 正上方。
- 保留原主校区四层结构。
- 曾尝试右移 AS1、AS2 及其下挂节点，因视觉效果不佳已撤回。

### 7.2 路径高亮与阻断叉

路径高亮根据 API 实际结果显示：

- 成功：绿色实线
- 阻断：红色路径、虚线衰减和阻断叉
- 公网入站、DMZ 横向移动和未开 VPN 的 `gst3` 均在 `fw1` 位置显示阻断

清理了以下错误展示：

- `gst3 → cs1` 虚假连线
- `gst3 → vpn_srv` 直接连线
- 外部网站挂在 DMZ 下方的视觉误导
- 无关 `cs1` 线路被错误高亮

### 7.3 DMZ/VPN 双向箭头

`sw15 ↔ vpn_srv` 使用两根平行方向箭头表示：

- 外层接入
- 解封装入网

`fw1 ↔ sw15` 也使用两根平行方向箭头：

- `fw1 → DMZ`
- `DMZ → fw1`

箭头采用与普通路径高亮一致的颜色逻辑：

- 未使用方向保持灰色虚线。
- 单向访问只点亮实际经过的方向。
- VPN 完整入网路径进出 DMZ 各一次，因此两个方向都会点亮。

原普通 `fw1-sw15` 线被隐藏，避免与双向箭头重叠。

### 7.4 分校区布局

分校区页面整体向上移动约 28px，减少底部节点和边框重叠。

## 9. 第二校区与链路指标

审查发现 `cs1-cs2` 当前使用普通：

```python
self.addLink(core_sw, cs2)
```

没有使用固定的 `TCLink`、`delay`、`loss` 或 `bw`。

因此移除了类似以下固定展示：

```text
50 Mbps / 18 ms / 0.2%
```

最终原则：

- 不把人工写死的参数描述为测量值。
- RTT、丢包、抖动来自实际 Ping。
- 吞吐来自接口计数器差分。
- 面板明确区分系统默认队列与已配置流量控制。

由于 Mininet 在本机运行，实际 RTT 可能很低，这是当前系统的真实结果，而不是模拟 WAN SLA。

## 14. 当前限制与演示口径

1. GRE 只提供隧道数据面，不提供生产 VPN 所需的加密、证书认证和密钥协商。
2. `fw1` 是课程环境中的 OVS 边界节点，实际有状态策略由 root namespace 的 Linux 路由和 `iptables` 执行。
3. 系统没有防火墙高可用、双机热备或多出口冗余。
4. 白名单基于 IP/服务目标，不是生产环境中的域名、证书或应用身份策略。
5. Mininet 后端必须完全重启，新的拓扑节点、OVS flow、iptables 和 veth trunk 才会生效。
6. 本轮已通过静态和逻辑回归测试，但最终演示前仍应在新版 Mininet 实例中执行一轮端到端场景测试。

## 15. 建议演示场景

按以下顺序可以清楚展示 DMZ 主创新点：

1. 外部网站访问普通办公终端：被 `fw1` 阻断。
2. 外部网站访问 DMZ 校园门户：成功。
3. 校园门户尝试访问内网：被 `fw1` 阻断，展示被入侵后的横向移动防护。
4. 宿舍或办公终端访问外部网站：成功。
5. 人事或财务访问外部网站：失败。
6. 人事访问白名单校园门户：成功。
7. `gst3` 未开 VPN 访问内网：在 `fw1` 阻断。
8. 开启 VPN 后再次访问普通内网：按完整 DMZ/VPN 路径成功。
9. VPN 访客访问人事或财务：仍被拒绝。
10. 展示观测面板中 RTT、接口吞吐、DMZ 探测和安全规则计数变化。

以上场景能够同时证明拓扑、实际转发、访问控制、VPN 和观测面板不是相互独立的静态展示，而是同一安全架构的不同表现层。
